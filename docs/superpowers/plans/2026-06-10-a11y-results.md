## What was fixed (this branch)

1. **Skip link had no target** on `/teacher-register` and `/forgot-password`
   (`skip-link` audit). The root layout renders `<a href="#main-content">` but
   those pages' `<main>` elements lacked the id. Added `id="main-content"` to
   both.
2. **Insufficient contrast (1.12:1)** on the `/forgot-password` mode toggle
   (`color-contrast` audit). The active tab hardcoded `bg-white` while text
   used `var(--ink-strong)`, which is near-white (`#f0f2f5`) under
   `[data-theme="dark"]` — near-white on white. Switched to the adaptive
   `bg-[var(--surface-raised)]` token.

## Honest scope notes

- **Authenticated pages were NOT measured.** The handoff allows axe checks via
  a seeded e2e login, but `scripts/seed-data.mjs` seeds only certification
  templates — no user credentials exist, and the existing e2e specs
  intentionally test the auth UI without a real login. Student/teacher
  dashboards, chat, and panels were built to AA tokens but remain unscored.
  To close this: seed a test user, then add `@axe-core/playwright` assertions
  inside `e2e/student-login-chat.spec.ts` / `e2e/teacher-dashboard.spec.ts`.
- **Same dark-mode `bg-white` pattern exists elsewhere**: 
  `src/components/auth/StaffMfaPanel.tsx` has several hardcoded `bg-white`
  containers whose text uses ink tokens. They render post-registration so no
  measured audit flagged them, but they will have the same dark-mode contrast
  problem. Worth a sweep for `bg-white` + `var(--ink-` combinations.
- Lighthouse automates only a subset of WCAG; manual checks (keyboard-only
  flows, screen-reader labels on the chat stream) were not part of this pass.
